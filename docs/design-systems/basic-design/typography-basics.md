# typography-basics.md

> Typography fundamentals for UI. Type scale, font weights, line height, letter spacing, font family, readability, and hierarchy. The rules here apply to every prototype regardless of design system; the M3-specific values live in `material-3-expressive/typography.md`.
>
> Read this before setting any `font-size`, `font-weight`, `line-height`, or `letter-spacing`.

---

## Why this file exists

Typography is the single largest contributor to a UI's perceived quality. Bad type — random sizes, inconsistent weights, lines too long, line-heights too tight — reads as "AI-generated" before the user parses a single word. Good type is invisible; the user just finds the screen easy to read.

AI agents consistently under-deliver on type because the defaults look "fine" in isolation. This file gives you the rules that make type hold up across a whole screen.

---

## The 7 sections (quick list)

1. **Type scale** — why use a scale, not random sizes.
2. **Font weights** — 400 / 500 / 600 / 700 and when to use each.
3. **Line height** — 1.5 for body, 1.2 for headlines.
4. **Letter spacing** — negative for large text, positive for small caps.
5. **Font family** — system stack for performance, Roboto for M3.
6. **Readability** — max line length 60–80 characters.
7. **Hierarchy through size + weight, not just size.**

---

## 1. Type scale

A type scale is a fixed set of font sizes, picked from a mathematical relationship (a **modular scale**), that you use for every text element on every screen. **Never invent a font size that isn't on the scale.**

### Why use a scale

1. **Visual rhythm.** The eye perceives type as a hierarchy of sizes. A scale produces a consistent rhythm; random sizes break it.
2. **Hierarchy clarity.** When sizes come from a scale, two elements at the same scale-step are unambiguously "the same level of importance." With random sizes, the user can't tell.
3. **Maintainability.** "Make the body text slightly bigger" means changing one CSS variable, not hunting through 30 rules.
4. **AI-tell avoidance.** Hand-tuned scales are a designer signature. Random sizes are an AI signature.

### Modular scales

A modular scale is generated by multiplying a base size by a ratio repeatedly:

| Ratio         | Multiplier | Example (base 14px)                    |
|---------------|------------|----------------------------------------|
| 1.125 (major second) | × 1.125 | 14, 15.75, 17.7, 19.9, 22.4, 25.2, 28.3 |
| 1.200 (minor third)  | × 1.2   | 14, 16.8, 20.2, 24.2, 29, 34.9          |
| 1.250 (major third)  | × 1.25  | 14, 17.5, 21.9, 27.3, 34.2              |
| 1.333 (perfect fourth) | × 1.333 | 14, 18.7, 24.9, 33.2                |
| 1.500 (perfect fifth) | × 1.5  | 14, 21, 31.5, 47.3 — large jumps, dramatic |

### The repo's default scale (M3)

This is the scale used in `material-3-expressive/typography.md`. Adopt it unless you have a specific reason not to.

```css
:root {
  --fs-display: 32px;    /* page-level hero / onboarding */
  --fs-h1:      26px;    /* page title */
  --fs-h2:      22px;    /* section title */
  --fs-h3:      18px;    /* card title, modal title */
  --fs-body-l:  16px;    /* emphasized body / lead paragraph */
  --fs-body:    14px;    /* default body */
  --fs-body-s:  13px;    /* dense lists, secondary body */
  --fs-label:   12px;    /* labels, captions, meta */
  --fs-label-s: 11px;    /* overlines, eyebrows (with uppercase + letter-spacing) */
}
```

This is a roughly 1.15-step scale, slightly compressed at the small end for UI density. It's not a pure modular scale — it's been tuned for UI legibility at small sizes.

### ✅ DO

```css
:root {
  --fs-display: 32px;
  --fs-h1:      26px;
  --fs-h2:      22px;
  --fs-h3:      18px;
  --fs-body:    14px;
  --fs-label:   12px;
}

.page-title    { font-size: var(--fs-h1); }
.section-title { font-size: var(--fs-h2); }
.card-title    { font-size: var(--fs-h3); }
.body          { font-size: var(--fs-body); }
.label         { font-size: var(--fs-label); }
```

### ❌ DON'T

```css
/* ❌ DON'T: arbitrary sizes, no system */
h1     { font-size: 36px; }
h2     { font-size: 24px; }
h3     { font-size: 20px; }
h4     { font-size: 18px; }
.x     { font-size: 17px; }
.y     { font-size: 15px; }
.body  { font-size: 14px; }
.label { font-size: 13px; }
.why   { font-size: 22px; }    /* what is this for? */
/* 8 sizes, no relationship, ambiguous hierarchy. */
```

### Don't go below 11px

Anything below 11px is unreadable for a significant fraction of users, especially on low-DPI screens. If you need text smaller than 11px, you don't need text — you need an icon, or you need to cut the content.

## 2. Font weights

Font weight is the second hierarchy lever (after size). Most UIs need four weights:

| Weight | Name       | Use                                                                |
|--------|------------|--------------------------------------------------------------------|
| 400    | Regular    | Default body text. The workhorse.                                  |
| 500    | Medium     | Buttons, labels, eyebrows, subtle emphasis without shouting.       |
| 600    | Semibold   | Card titles, section titles, emphasis where bold would be too much. |
| 700    | Bold       | Page titles, hero numbers, strong emphasis.                        |

### When to use each

- **400 (regular):** body text, captions, descriptions, anything meant to be read at length.
- **500 (medium):** UI labels (button text, tab labels, chip text), eyebrows (uppercase overlines), meta text where you want it slightly more present than body but not bold.
- **600 (semibold):** titles of cards, sections, modals. The default "this is a heading" weight in most modern UIs.
- **700 (bold):** the page title, the hero stat, the single most important text on the screen. Use sparingly.

### ✅ DO

```css
:root {
  --fw-regular:  400;
  --fw-medium:   500;
  --fw-semibold: 600;
  --fw-bold:     700;
}

.page-title    { font-size: var(--fs-h1);   font-weight: var(--fw-bold); }     /* 26px / 700 */
.section-title { font-size: var(--fs-h2);   font-weight: var(--fw-semibold); } /* 22px / 600 */
.card-title    { font-size: var(--fs-h3);   font-weight: var(--fw-semibold); } /* 18px / 600 */
.body          { font-size: var(--fs-body); font-weight: var(--fw-regular); }  /* 14px / 400 */
.label         { font-size: var(--fs-label); font-weight: var(--fw-medium); }  /* 12px / 500 */
.btn           { font-size: var(--fs-body); font-weight: var(--fw-medium); }   /* 14px / 500 */
.eyebrow       { font-size: var(--fs-label-s); font-weight: var(--fw-medium);
                 text-transform: uppercase; letter-spacing: 0.08em; }           /* 11px / 500 */
```

### ❌ DON'T

```css
/* ❌ DON'T: only one weight, only one lever */
* { font-weight: 400; }
.page-title { font-size: 32px; }   /* hierarchy via size only */
.section-title { font-size: 26px; }
.card-title { font-size: 22px; }
/* Headings are bigger but feel flat — no contrast in weight. */

/* ❌ DON'T: 300 (light) weight for body */
body { font-weight: 300; }
/* Light weights are unreadable at small sizes on low-DPI screens.
   Reserve 300 for display-sized hero text, not body. */

/* ❌ DON'T: 800/900 (extra-bold/black) for UI */
.btn { font-weight: 800; }
/* Too heavy for UI; reads as 90s Microsoft Word. */
```

### Font-loading note

If you're using a webfont, **load only the weights you use.** Each weight is a separate font file (~20–80KB). Loading 4 weights when you only use 2 wastes bandwidth.

```css
/* ✅ DO: only the weights you need */
@font-face {
  font-family: 'Roboto';
  font-weight: 400;
  src: url('Roboto-Regular.woff2') format('woff2');
  font-display: swap;   /* show fallback immediately, swap when loaded */
}
@font-face {
  font-family: 'Roboto';
  font-weight: 500;
  src: url('Roboto-Medium.woff2') format('woff2');
  font-display: swap;
}
/* 600 and 700 only if you use them. */
```

## 3. Line height

Line height (leading) is the vertical space between lines of text. It's the most under-appreciated typographic variable. Too tight: text feels cramped, scanning is hard. Too loose: text feels airy but paragraphs lose cohesion.

### The rules of thumb

| Text type              | Line height | Why                                            |
|------------------------|-------------|------------------------------------------------|
| Body text              | **1.5**     | Comfortable for multi-line reading. WCAG 1.4.12 minimum. |
| Headlines (≥ 22px)     | **1.2–1.3** | Tight lines make headlines feel like a unit; loose lines split them. |
| Display text (≥ 32px)  | **1.1–1.2** | Even tighter; large text doesn't need extra leading. |
| UI labels (≤ 14px, single line) | 1.0–1.2 | Single-line text doesn't need leading; tight = compact. |
| Multi-line small text (12–14px) | 1.4–1.5 | Slightly tighter than body since the text is smaller. |

### ✅ DO

```css
:root {
  --lh-display: 1.15;
  --lh-heading: 1.25;
  --lh-body:    1.5;
  --lh-tight:   1.2;
  --lh-label:   1.0;   /* single-line labels */
}

.display { font-size: var(--fs-display); line-height: var(--lh-display); } /* 32px / 1.15 */
.h1      { font-size: var(--fs-h1);      line-height: var(--lh-heading); } /* 26px / 1.25 */
.h2      { font-size: var(--fs-h2);      line-height: var(--lh-heading); } /* 22px / 1.25 */
.h3      { font-size: var(--fs-h3);      line-height: var(--lh-tight); }    /* 18px / 1.2 */
.body    { font-size: var(--fs-body);    line-height: var(--lh-body); }     /* 14px / 1.5 */
.label   { font-size: var(--fs-label);   line-height: var(--lh-label); }    /* 12px / 1.0 */
```

### ❌ DON'T

```css
/* ❌ DON'T: default line-height (1.0–1.2) on body text */
.body { font-size: 14px; line-height: 1.2; }
/* Multi-line body text at 1.2 line-height is cramped; reading speed drops ~15%. */

/* ❌ DON'T: same line-height for everything */
* { line-height: 1.5; }
.h1 { font-size: 32px; line-height: 1.5; }
/* A 32px headline at 1.5 line-height has 16px between lines — looks loose and split. */

/* ❌ DON'T: line-height in px (or any unit other than unitless) */
.body { line-height: 21px; }
/* If font-size changes (e.g. user zooms, or the text inherits a different size),
   the px line-height doesn't scale — text gets cramped or loose. */
```

### Why unitless line-height

Use unitless line-height (`1.5`, not `1.5em` or `24px`). Unitless line-height is **relative to the element's own font-size**, so it scales correctly when font-size changes (via inheritance, media queries, or user zoom). `1.5em` is relative to the *parent's* font-size, which is rarely what you want.

```css
/* ✅ DO */   .body { line-height: 1.5; }
/* ❌ DON'T */.body { line-height: 1.5em; }   /* relative to parent — surprise inheritance */
/* ❌ DON'T */.body { line-height: 24px; }    /* doesn't scale with font-size */
```

## 4. Letter spacing

Letter spacing (tracking) is the horizontal space between characters. Two rules: **negative for large text, positive for small caps.**

### The rules

| Text type                          | Letter spacing    | Why                                              |
|------------------------------------|--------------------|--------------------------------------------------|
| Display (≥ 28px)                   | **−0.02em to −0.04em** | Large text has natural optical spacing that makes letters feel too far apart. Tighten. |
| Headlines (18–26px)                | **−0.005em to −0.015em** | Mild tightening for cohesion.                  |
| Body (14–16px)                     | **0 (normal)**     | Don't touch. Body text is designed to be read at normal tracking. |
| Small text (≤ 13px)                | **0 to +0.01em**   | Very slight opening helps legibility at small sizes. |
| All-caps labels / overlines (≤ 12px) | **+0.04em to +0.12em** | All-caps text needs extra tracking to be legible; the tight default looks cramped. |

### ✅ DO

```css
:root {
  --ls-display: -0.02em;
  --ls-h1:      -0.015em;
  --ls-h2:      -0.01em;
  --ls-h3:      -0.005em;
  --ls-body:    0;
  --ls-label:   0;
  --ls-overline: 0.08em;
}

.display  { font-size: var(--fs-display); letter-spacing: var(--ls-display); }
.h1       { font-size: var(--fs-h1);      letter-spacing: var(--ls-h1); }
.h2       { font-size: var(--fs-h2);      letter-spacing: var(--ls-h2); }
.body     { font-size: var(--fs-body);    letter-spacing: var(--ls-body); }
.overline { font-size: var(--fs-label-s); font-weight: var(--fw-medium);
            text-transform: uppercase; letter-spacing: var(--ls-overline); }
```

### ❌ DON'T

```css
/* ❌ DON'T: positive letter-spacing on large text */
.h1 { font-size: 32px; letter-spacing: 0.05em; }
/* Looks airy and amateurish — like a 90s band flyer. */

/* ❌ DON'T: leave all-caps at default tracking */
.eyebrow { text-transform: uppercase; }   /* no letter-spacing */
/* 'SECTIONTITLE' — letters crash into each other. Needs +0.08em. */

/* ❌ DON'T: letter-spacing on body text */
.body { letter-spacing: 0.02em; }
/* Looks like a ransom note. Body text is designed at 0 tracking. */
```

### Why this matters

Letter spacing is a subtle but powerful signal. Tightened large text reads as "designed." Default-tracking body reads as "considered." Properly tracked all-caps reads as "intentional." Getting these wrong is one of the AI-tells that scream "I didn't think about the type."

## 5. Font family

For UI, you want a font that:

1. **Renders well at small sizes** (11–14px) on low-DPI screens. Many beautiful display fonts fail here.
2. **Has the weights you need** (400, 500, 600, 700 at minimum).
3. **Loads fast** (or is already on the user's device).
4. **Has good Cyrillic / Greek / CJK coverage** if you need it.

### The system stack (default for non-M3 prototypes)

For maximum performance, use the OS's default UI font. It's already on the user's device — zero download, zero FOUT (flash of unstyled text), zero layout shift.

```css
:root {
  --font-sans: system-ui, -apple-system, 'Segoe UI', Roboto, 'Helvetica Neue', Arial, sans-serif;
  --font-mono: ui-monospace, 'SF Mono', 'Cascadia Code', 'Roboto Mono', Consolas, monospace;
}

body { font-family: var(--font-sans); }
code, pre { font-family: var(--font-mono); }
```

This resolves to:
- **macOS / iOS:** San Francisco (SF Pro).
- **Windows:** Segoe UI.
- **Android:** Roboto.
- **Linux:** depends on the distro; usually a sans-serif fallback.

All four are excellent UI fonts. The system stack is the right default.

### Roboto (M3 prototypes)

Material 3 specifies Roboto. For M3 prototypes, load Roboto explicitly so the prototype looks the same on all devices (otherwise iOS users get San Francisco, which is close but not identical).

```css
@import url('https://fonts.googleapis.com/css2?family=Roboto:wght@400;500;700&display=swap');

:root {
  --font-sans: 'Roboto', system-ui, -apple-system, sans-serif;
}
```

Or self-host for performance and offline support:

```css
@font-face {
  font-family: 'Roboto';
  font-weight: 400;
  src: url('/fonts/Roboto-Regular.woff2') format('woff2');
  font-display: swap;
}
/* repeat for 500, 700 */
```

See `material-3-expressive/typography.md` for the M3-specific stack.

### Display fonts (use sparingly)

If your brand calls for a distinctive display font (serif, geometric, etc.), use it **only for headings**. Keep body text in the system stack or a proven UI font. Display fonts almost always render poorly at body sizes.

```css
:root {
  --font-display: 'Fraunces', Georgia, serif;
  --font-body:    system-ui, -apple-system, sans-serif;
}

.h1, .h2, .display { font-family: var(--font-display); }
body { font-family: var(--font-body); }
```

### ❌ DON'T

```css
/* ❌ DON'T: load a webfont without font-display: swap */
@font-face {
  font-family: 'Roboto';
  src: url('Roboto-Regular.woff2') format('woff2');
  /* no font-display → defaults to block → invisible text for up to 3s while loading */
}

/* ❌ DON'T: use a display font for body */
body { font-family: 'Playfair Display', serif; }
/* Playfair is beautiful at 48px; at 14px it's a fuzzy mess. */

/* ❌ DON'T: use Comic Sans, Papyrus, or any "novelty" font. Ever. */
/* (You knew this. Listed for completeness.) */

/* ❌ DON'T: ship Inter as a webfont when system-ui would do. */
/* Inter is great but if all your users have San Francisco / Segoe UI / Roboto on their devices,
   loading Inter is 60KB of wasted bandwidth for a near-identical result. */
```

## 6. Readability

Readability is about the *paragraph* experience, not the individual letters. Two key variables:

### Line length (measure)

The optimal line length for body text is **50–75 characters** (including spaces). The hard limits are **45 minimum, 85 maximum.** Beyond 85, the eye has trouble finding the start of the next line; below 45, the eye jumps too often and reading is choppy.

### ✅ DO

```css
.prose {
  max-width: 65ch;        /* 65 characters — the goldilocks zone */
  font-size: var(--fs-body);
  line-height: var(--lh-body);
}
```

The `ch` unit is the width of the `0` character in the current font — using `max-width: 65ch` makes the line length self-adjusting to the font.

### ❌ DON'T

```css
/* ❌ DON'T: full-width body text on desktop */
.prose { width: 100%; }
/* On a 1920px monitor at 16px font, that's ~190 characters per line.
   The eye gets lost returning to the next line; reading speed drops ~30%. */

/* ❌ DON'T: super-narrow text columns */
.prose { max-width: 30ch; }
/* 30 characters per line — eye jumps every 2 seconds. Annoying. */
```

### Line height (see §3)

Line height and line length interact: longer lines need looser line height to help the eye find the next line. Shorter lines can be tighter. The 1.5 body / 1.2 headline defaults work well at the 65ch sweet spot.

### Justification

**Don't justify text on the web.** Justification requires hyphenation, and CSS hyphenation is inconsistent across browsers. The result is "rivers" of white space running down the page.

```css
/* ✅ DO */   .prose { text-align: left; }     /* or 'start' for RTL support */
/* ❌ DON'T */.prose { text-align: justify; }   /* rivers of whitespace */
```

### Hyphenation (if you must)

```css
.prose {
  hyphens: auto;
  -webkit-hyphens: auto;
}
/* Only enable if you've tested across browsers and the lang attribute is set on the html element
   (hyphenation dictionaries are language-specific). */
```

```html
<html lang="en">
```

## 7. Hierarchy through size + weight, not just size

The biggest typographic mistake AI makes: building hierarchy with size alone. A 32px regular headline next to a 14px regular body looks flat — the headline is bigger but it doesn't feel more important. Add weight (and color, and letter-spacing) to make the hierarchy sing.

### The hierarchy recipe

For each level of the hierarchy, change **at least two** of: size, weight, color, letter-spacing.

| Level   | Size  | Weight | Color                | Letter-spacing |
|---------|-------|--------|----------------------|----------------|
| Display | 32px  | 700    | `--on-surface`       | −0.02em        |
| H1      | 26px  | 700    | `--on-surface`       | −0.015em       |
| H2      | 22px  | 600    | `--on-surface`       | −0.01em        |
| H3      | 18px  | 600    | `--on-surface`       | −0.005em       |
| Body    | 14px  | 400    | `--on-surface-variant` | 0            |
| Label   | 12px  | 500    | `--on-surface-variant` | 0            |
| Overline| 11px  | 500    | `--primary`          | +0.08em (uppercase) |

### ✅ DO

```css
.page-title {
  font-size: var(--fs-h1);           /* 26px */
  font-weight: var(--fw-bold);       /* 700 */
  color: var(--on-surface);
  letter-spacing: var(--ls-h1);
  line-height: var(--lh-heading);
}
.section-title {
  font-size: var(--fs-h2);           /* 22px */
  font-weight: var(--fw-semibold);   /* 600 */
  color: var(--on-surface);
  letter-spacing: var(--ls-h2);
  line-height: var(--lh-heading);
}
.card-title {
  font-size: var(--fs-h3);           /* 18px */
  font-weight: var(--fw-semibold);   /* 600 */
  color: var(--on-surface);
  letter-spacing: var(--ls-h3);
  line-height: var(--lh-tight);
}
.body {
  font-size: var(--fs-body);         /* 14px */
  font-weight: var(--fw-regular);    /* 400 */
  color: var(--on-surface-variant);
  line-height: var(--lh-body);
}
.eyebrow {
  font-size: var(--fs-label-s);      /* 11px */
  font-weight: var(--fw-medium);     /* 500 */
  color: var(--primary);
  text-transform: uppercase;
  letter-spacing: var(--ls-overline);
  line-height: var(--lh-label);
}
```

### ❌ DON'T

```css
/* ❌ DON'T: hierarchy via size only */
.page-title    { font-size: 26px; }
.section-title { font-size: 22px; }
.card-title    { font-size: 18px; }
.body          { font-size: 14px; }
/* All weight 400, all same color. Hierarchy is weak;
   the page reads as a flat list of slightly different sizes. */

/* ❌ DON'T: hierarchy via weight only (no size change) */
.page-title    { font-weight: 700; }
.section-title { font-weight: 600; }
.card-title    { font-weight: 600; }
.body          { font-weight: 400; }
/* All 14px. The hierarchy is there but the page reads as one block of text. */
```

### The compound effect

When you change size + weight + color + letter-spacing together, each level of the hierarchy becomes unambiguous. The user can scan the page in <1 second and know what's a title, what's a section, what's a card, what's body. That's the goal.

## Putting it all together: the typographic boilerplate

```css
:root {
  /* Type scale (M3 default) */
  --fs-display: 32px;
  --fs-h1:      26px;
  --fs-h2:      22px;
  --fs-h3:      18px;
  --fs-body-l:  16px;
  --fs-body:    14px;
  --fs-body-s:  13px;
  --fs-label:   12px;
  --fs-label-s: 11px;

  /* Weights */
  --fw-regular:  400;
  --fw-medium:   500;
  --fw-semibold: 600;
  --fw-bold:     700;

  /* Line heights */
  --lh-display: 1.15;
  --lh-heading: 1.25;
  --lh-body:    1.5;
  --lh-tight:   1.2;
  --lh-label:   1.0;

  /* Letter spacing */
  --ls-display: -0.02em;
  --ls-h1:      -0.015em;
  --ls-h2:      -0.01em;
  --ls-h3:      -0.005em;
  --ls-body:    0;
  --ls-overline: 0.08em;

  /* Font family */
  --font-sans: system-ui, -apple-system, 'Segoe UI', Roboto, 'Helvetica Neue', Arial, sans-serif;
  --font-mono: ui-monospace, 'SF Mono', 'Cascadia Code', 'Roboto Mono', Consolas, monospace;
}

body {
  font-family: var(--font-sans);
  font-size: var(--fs-body);
  font-weight: var(--fw-regular);
  line-height: var(--lh-body);
  color: var(--on-surface);
  -webkit-font-smoothing: antialiased;
  -moz-osx-font-smoothing: grayscale;
}

/* Headings */
.display  { font-size: var(--fs-display); font-weight: var(--fw-bold);
            line-height: var(--lh-display); letter-spacing: var(--ls-display); }
.h1       { font-size: var(--fs-h1);      font-weight: var(--fw-bold);
            line-height: var(--lh-heading); letter-spacing: var(--ls-h1); }
.h2       { font-size: var(--fs-h2);      font-weight: var(--fw-semibold);
            line-height: var(--lh-heading); letter-spacing: var(--ls-h2); }
.h3       { font-size: var(--fs-h3);      font-weight: var(--fw-semibold);
            line-height: var(--lh-tight);   letter-spacing: var(--ls-h3); }

/* Body */
.body     { font-size: var(--fs-body); font-weight: var(--fw-regular);
            line-height: var(--lh-body); letter-spacing: var(--ls-body);
            color: var(--on-surface-variant); }
.body-l   { font-size: var(--fs-body-l); font-weight: var(--fw-regular);
            line-height: var(--lh-body); }

/* UI labels */
.label    { font-size: var(--fs-label); font-weight: var(--fw-medium);
            line-height: var(--lh-label); }
.eyebrow  { font-size: var(--fs-label-s); font-weight: var(--fw-medium);
            text-transform: uppercase; letter-spacing: var(--ls-overline);
            color: var(--primary); }

/* Buttons */
.btn      { font-size: var(--fs-body); font-weight: var(--fw-medium);
            line-height: var(--lh-label); letter-spacing: var(--ls-body); }

/* Prose (long-form reading) */
.prose    { max-width: 65ch;
            font-size: var(--fs-body-l);   /* slightly larger for reading */
            line-height: var(--lh-body);
            color: var(--on-surface); }
.prose p  { margin-bottom: 1em; }
```

Drop this into a project, use only the defined classes / tokens, and the typography will hold up.

## The typography verification checklist

After generating any UI, walk through these:

1. **Scale:** Are all font sizes from a defined scale? Are there any "orphan" sizes like 15px or 17px?
2. **Weights:** Are at least 3 weights in use (400, 500/600, 700)? Is 400 used for body, 600/700 for headings?
3. **Line height:** Is body at 1.5? Are headlines at 1.2–1.25? Is line-height unitless?
4. **Letter spacing:** Are large headlines tightened (negative)? Are all-caps eyebrows loosened (positive)?
5. **Font family:** Is the system stack or a proven UI font in use? Are webfonts loaded with `font-display: swap`? Are only the weights you use loaded?
6. **Readability:** Is body text capped at 65ch? Is text left-aligned (not justified)?
7. **Hierarchy:** Does each level of the hierarchy differ in at least 2 of (size, weight, color, letter-spacing)?

If any answer is "no," fix it before shipping.
